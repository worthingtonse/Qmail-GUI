4. FEATURES WITH NO GUI (Backend exists, no UI)
========================================================================

4.1  Locker Pool Status  [NEW - NO GUI]
     - Backend: GET /api/qmail/locker-pool (just implemented)
     - Returns: upload_available, inbox_fee_available, wallet_balance,
       low_balance
     - No GUI component displays pool status
     - No warning shown when low_balance_warning=true
     - Phase 2 candidate: Add pool status indicator to ComposeModal
       or AccountPane

4.2  Send Response Warnings  [NEW - NOT CONSUMED BY GUI]
     - Backend returns used_sync_fallback and low_balance_warning
       in send response JSON
     - GUI does not read or display these fields
     - User won't know if their sends are slow (sync fallback) or
       if they're running low on coins
     - Phase 2: Show warning toast after send if low_balance_warning

4.3  Inbox Fee Lookup  [NEW - NO GUI]
     - Backend: GET /api/qmail/inbox-fee?sn=<serial_number>
     - Created for beacon server use
     - GUI could show a contact's inbox fee in ContactsPane or
       ComposeModal (e.g., "Sending to @alice costs 5 CC")
     - Phase 2 candidate

4.4  Payment Claim  [EXISTS IN BACKEND, NO GUI]
     - Backend: GET /api/qmail/payment?email_id=<hex>
     - Backend: POST /api/qmail/payment/claim?email_id=<hex>
     - When user receives email with payment locker, they can
       claim the coins
     - No GUI button or flow exists to claim payment
     - Phase 2 candidate: Add "Claim Payment" button in ReadingPane
       when inboxFee > 0

4.5  Inbox Fee Display  [DATA STORED, NOT SHOWN]
     - inboxFee field is extracted from API responses:
       qmailApiServices.js:175, QMailDashboard.jsx:458
     - Stored in email objects but never rendered in ReadingPane
       or EmailListItem
     - Phase 2: Show fee badge on emails with payment

4.6  Email Heal Identity  [API EXISTS, MINIMAL GUI]
     - Backend: POST /api/qmail/heal-identity
     - Called from qmailApiServices.js healWallet()
     - No prominent UI trigger -- may only be in WalletSetupScreen

4.7  Prepare Change  [API CALL EXISTS, UNCLEAR INTEGRATION]
     - prepareChange() in qmailApiServices.js
     - Calls GET /api/coins/prepare-change
     - Used in ComposeModal send flow but actual integration unclear

========================================================================
6. OBSERVATIONS
========================================================================

6.1  API Migration History
     Many "API-FIX" comments show endpoints were recently migrated:
     - /api/mail/* -> /api/qmail/*
     - /api/wallets/active/balance -> /api/wallets/balance
     - /api/task/status -> /api/system/tasks
     Some old patterns may linger.

6.2  Sound System
     - soundService.js and useGlassSounds.js exist
     - Glass click/tap sounds integrated into wallet MainDashboard
     - Only one sound file: public/sounds/ding-80828.mp3
     - Unclear if new mail notification sound is implemented

6.3  Electron IPC vs REST
     - Some operations use Electron IPC (file reads, USB check)
     - Most use REST API calls to localhost:8080
     - BUG-10 FIX shows migration from IPC to REST for some calls

6.4  Error Handling Pattern
     - Consistent {success, data, error} response structure
     - handleResponse() in apiService.js for standard error handling
     - Notification system (NotificationContext) for user-facing errors

6.5  Async Task Polling
     - Long operations return task_id, GUI polls via getTaskStatus()
     - pollTaskUntilComplete() with configurable interval and callback
     - Used for: send, import, export, locker operations, health checks
