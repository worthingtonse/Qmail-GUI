BUG-01 — Path Traversal in read-file IPC handler
  File:     electron.cjs, lines 175-183
  Severity: CRITICAL
  Found by: Claude, Gemini (partial — BUG-13 noted dev-mode path issue)

  The 'read-file' IPC handler joins process.resourcesPath with the
  caller-supplied filename without any validation or sanitization:

    ipcMain.handle('read-file', async (event, filename) => {
      const filePath = path.join(process.resourcesPath, filename);
      const content = fs.readFileSync(filePath, 'utf-8');
      return { success: true, content };
    });

  Any renderer-side code can pass '../../../../../../etc/passwd' or any
  other relative path and the main process will read and return its
  contents. Even though contextIsolation is enabled, any code in the
  renderer can call window.electronAPI.readFile(maliciousPath).

  Additionally (from Gemini BUG-13): In dev mode, process.resourcesPath
  is undefined or points to Electron's internal resources, not the app's
  public/ folder. So readFile always fails in development.

  FIX PLAN:
  1. Validate that the resolved path stays within a safe base directory:
       const base = isDev ? path.join(__dirname, 'public') : process.resourcesPath;
       const resolved = path.resolve(base, filename);
       if (!resolved.startsWith(base)) {
         return { success: false, error: 'Access denied' };
       }
  2. Also whitelist allowed filenames (currently only eff_large_wordlist.txt
     is used) to further reduce attack surface.

       const freshDrafts = await loadDrafts();

BUG-04 — Infinite polling loop in ComposeModal with no cancellation
  File:     src/qmail/screens/ComposeModal.jsx, lines 486-525
  Severity: HIGH
  Found by: Claude, GPT (related)

  handleSend runs a while(true) loop polling getTaskStatus with 1-second
  delays. There is no way to cancel this loop. If the user closes the
  modal or navigates away while polling, the loop continues calling
  setState on the now-unmounted component (memory leak + React warnings).
  It can also call onSend(emailData) via setTimeout after the component
  is gone.

  FIX PLAN:
  1. Add a useRef for cancellation: const cancelledRef = useRef(false);
  2. In the useEffect cleanup, set cancelledRef.current = true.
  3. Check cancelledRef.current at the top of each loop iteration:
       while (!taskFinished && !cancelledRef.current) { ... }
  4. Guard the setTimeout callbacks with the same check.

  File:     src/api/apiService.js, lines 508-575
  Severity: HIGH

  FIX PLAN:

  2. If the backend truly has CORS issues on DELETE, fix it server-side

BUG-07 — pollTaskUntilComplete loops forever on unknown task status
  File:     src/api/apiService.js, lines 838-873
  Severity: HIGH
  Found by: Claude

  pollTaskUntilComplete runs while(true) and only breaks on status
  'success', 'error', or 'failed'. If the backend returns any other
  terminal status (e.g., 'cancelled', 'timeout', 'aborted'), the loop
  never exits. The UI is permanently stuck with isProcessing = true.

  FIX PLAN:

  2. Add a catch-all for unknown statuses after N attempts.
  3. Handle additional terminal states: 'cancelled', 'timeout', 'aborted'.

BUG-08 — %USERNAME% placeholder never expanded in expandPath
  File:     src/api/apiService.js, lines 38-57
  Severity: HIGH
  Found by: Claude, Gemini (BUG-1)

  The commonExpansions map uses literal %USERNAME% strings:

    'Downloads': 'C:\\Users\\%USERNAME%\\Downloads',

  JavaScript does not expand Windows environment variables. The resulting
  paths sent to the backend will literally contain '%USERNAME%', causing
  all import/export operations using short folder names to fail.

  FIX PLAN:
  1. Add an IPC call to get the home directory from the main process:
       // In electron.cjs:
       ipcMain.handle('get-home-dir', () => require('os').homedir());
       // In preload.cjs:
       getHomeDir: () => ipcRenderer.invoke('get-home-dir')
  2. In apiService.js, call this at initialization and use the result
     to build the commonExpansions map dynamically.
  3. Alternatively, let the backend resolve paths (it has access to env vars).

BUG-09 — Backend respawned on every window refresh (F5)
  File:     electron.cjs, lines 135-138
  Severity: HIGH
  Found by: Gemini (BUG-11)

  startBackend() is called inside the 'did-finish-load' event:

    mainWindow.webContents.on('did-finish-load', () => {
      startBackend();
    });

  This event fires every time the page reloads (F5, location.reload()).
  Each reload spawns a new qmail-backend.exe without killing the previous
  one. This causes port 8080 conflicts and orphaned processes.

  FIX PLAN:
  1. Guard startBackend() to only spawn if backendProcess is null:
       mainWindow.webContents.on('did-finish-load', () => {
         if (!backendProcess) startBackend();
       });
  2. Or better: move startBackend() to app.whenReady(), before window
     creation, so it only runs once.

BUG-10 — USB check IPC returns object, but frontend expects boolean
  File:     electron.cjs, line 149; src/wallet/Wallet.jsx, line 30
  Severity: HIGH
  Found by: Gemini (BUG-12), GPT (related)

  The IPC handler returns { success: true } (an object):

    ipcMain.handle('check-usb-drive', async () => {
      return { success: true };
    });

  But Wallet.jsx sets state directly with the result:

    const result = await window.electronAPI.checkUSBDrive();
    setIsUSBDrive(result);  // result is { success: true }, not a boolean

  Since any object is truthy, isUSBDrive is always "truthy" regardless of
  actual USB status. The USB check is effectively bypassed.

  Additionally (from GPT): USBCheckScreen copies the prop into local state
  via useState(isUSBDrive) on mount, so even if the parent updates the
  prop later, the local state never changes.

  FIX PLAN:
  1. Fix the IPC handler to return a boolean:
       ipcMain.handle('check-usb-drive', async () => {
         // Actual USB detection logic here
         return true; // or false
       });
  2. In USBCheckScreen, either remove the local state copy and use the
     prop directly, or sync it via useEffect when the prop changes.

BUG-21 — Race condition: folder switch without request cancellation
  File:     src/qmail/screens/QMailDashboard.jsx, lines 266-346, 348-361
  Severity: MEDIUM
  Found by: Gemini (BUG-3)

  loadEmails has no cancellation or "last-request-wins" tracking. If a
  user quickly switches between folders, multiple fetch requests run
  concurrently. The last one to resolve overwrites setEmails(), but that
  may not correspond to the currently selected folder.

  FIX PLAN:
  1. Use an AbortController stored in a ref. Abort the previous request
     when a new loadEmails call begins:
       const abortRef = useRef(null);
       const loadEmails = async (folder) => {
         if (abortRef.current) abortRef.current.abort();
         const controller = new AbortController();
         abortRef.current = controller;
         const result = await getMailList(folder, ..., { signal: controller.signal });
         ...
       };

  PART 2 — REJECTED BUGS FROM OTHER MODELS
###############################################################################

The following bugs reported by Gemini and GPT were reviewed and rejected.
Reasons are given for each.

===============================================================================

REJECTED-01 — Gemini BUG-4: "Inconsistent API Implementation" (two
  handleResponse functions)
  Reason: This is a code quality/maintainability observation, not a bug.
  Having two files with different handleResponse implementations is a
  design choice (apiService.js for wallet operations, qmailApiServices.js
  for email operations). Both work correctly for their respective use
  cases. The different error handling strategies are intentional — wallet
  operations are more lenient, email operations extract backend error
  messages. No user-facing bug results from this.

-------------------------------------------------------------------------------

REJECTED-02 — Gemini BUG-5: "Hardcoded API_BASE_URL"
  Reason: Not a bug. This is a desktop Electron app that always runs its
  own local backend on port 8080. There is no production/staging
  environment distinction. The backend is bundled with the app. Hardcoding
  localhost:8080 is the correct design for this use case.

-------------------------------------------------------------------------------

REJECTED-03 — Gemini BUG-7: "Inefficient client-side filtering in
  getContacts"
  Reason: Performance observation, not a bug. For a personal email client,
  the contact list is unlikely to grow large enough for this to matter.
  The filtering works correctly. If performance becomes an issue, the
  backend search endpoint could be used instead, but this is an
  optimization, not a bug fix.

-------------------------------------------------------------------------------

REJECTED-04 — Gemini BUG-8: "Manual path normalization is error-prone"
  Reason: Code quality observation, not a bug. The normalizePath function
  works correctly with its boolean flag. The concern that developers
  might use the wrong flag is a maintainability issue, not a runtime bug.
  No incorrect behavior has been identified.

-------------------------------------------------------------------------------

REJECTED-05 — GPT BUG-3: "pollTaskStatus reads result.data.status but
  getTaskStatus returns state"
  Reason: While the field mismatch is real (status vs state), the
  pollTaskStatus function is dead code. The old handleSend that called it
  (lines 350-437) is entirely commented out. The active handleSend
  (line 439) does inline polling and correctly destructures 'state',
  'isFinished', 'isSuccessful' from result.data. No runtime impact.
  If pollTaskStatus is ever re-enabled, the field mismatch would need
  fixing, but as shipped, this is not a bug.

-------------------------------------------------------------------------------

REJECTED-06 — Gemini BUG-3 (partial): "Multiple setInterval loops cause
  redundant re-renders"
  Reason: Having multiple intervals (10s for notifications, 60s for mail
  counts, 120s for wallet balance) is not a bug. They update different
  state variables at different frequencies, which is intentional. The
  re-renders they cause are normal React behavior and do not constitute
  a bug. The race condition aspect of folder switching WAS accepted as
  BUG-21.
